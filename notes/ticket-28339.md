# Ticket 28339 - Expand the docs around how sqlmigrate works with regards to constraint and index names

- **Branch:** `ticket-28339`
- **Date:** 2026-07-18
- **URL:** https://code.djangoproject.com/ticket/28339

## Summary

When there is a migration that creates an index/constraint on a column followed by another migration that removes the index/constraint, Django will generate the SQL statement based on the state of the database.

That is, if the database state shows the index does not currently exist, then Django does not generate a sql statement to drop the index.

On the other hand, if the index does exist, then Django will introspect the database to get the name of the index/constraint and generate the sql statement to drop them.

Ticket was repurposed to update documentation for sqlmigrate.

## Vocabulary
- sqlmigrate - django command that shows the sql statement generated for a migration
- db_index - the keyword for including/not including a db index on a column. FK will by default create an index on the column.
- introspection - Django connects to the database to discern the state of the database.
  - Last week, from looking at ticket-25243, we learned about the `inspectdb` command, which connects to the database to read the tables and prints out the corresponding Django models.


### Reproduction (running against sqlite)
1. Make a field with db_index=True
2. Make migration. Check sqlmigrate.

```sql
BEGIN;
--
-- Alter field name on book
--
CREATE TABLE "new__books_book" ("id" integer NOT NULL PRIMARY KEY AUTOINCREMENT, "name" varchar(255) NOT NULL);
INSERT INTO "new__books_book" ("id", "name") SELECT "id", "name" FROM "books_book";
DROP TABLE "books_book";
ALTER TABLE "new__books_book" RENAME TO "books_book";
CREATE INDEX "books_book_name_f0f4e83a" ON "books_book" ("name");
COMMIT;
```

3. Explicitly set db_index=False. Create migration.
4. Check sqlmigrate. No DROP INDEX statement.

```sql
BEGIN;
--
-- Alter field name on book
--
CREATE TABLE "new__books_book" ("id" integer NOT NULL PRIMARY KEY AUTOINCREMENT, "name" varchar(255) NOT NULL);
INSERT INTO "new__books_book" ("id", "name") SELECT "id", "name" FROM "books_book";
DROP TABLE "books_book";
ALTER TABLE "new__books_book" RENAME TO "books_book";
COMMIT;
```

### Notes
Bug reporter was running against postgres, while I am running against sqlite. Interesting how sqlite copies data from old table to new table.

- Migration 0001, table is created with name "books_book".
- Migration 0002,
	1. Table name changed to "new__books_book".
	2. Data is transferred from "books_book" to the new table.
	3. Old table is dropped.
	4. New table renamed to have old table's name.
	5. Index applied on new table.
- Migration 0003
	1. Table name changed to "new__books_book".
	2. Data transferred from "books_book" to new table.
	3. Old table is dropped.
	4. New table renamed to have old table's name.
	5. No further operations. (No index applied to new table.)


## Ticket Comments
- Django doesn't know the index name ahead of time.
- How Django names the constraints and indexes has changed between versions.
- If you create the migration files without actually running the migrations against the database, then sqlmigrate doesn't actually create the index.
- If you run the migration from step 1 and then run the migration from step 2 in two separate steps, then sqlmigrate will create the index then drop it (although with sqlite we see it creates a new table without the index and moves data into it). In this case, Django will look into the database determine the names of the constraints and indexes it needs to add/drop.
- Bug reporter cannot "convert autogenerated django migrations to SQL up-front."

## Questions

- Are there other areas of migration besides index/constraints where sqlmigrate will generate different sql statements depending on db state?

I wasn't too sure if the last question was valid, so I consulted DeepSeek V4 Flash:

The existing docs cover constraint/index name resolution, but there are other areas where sqlmigrate output varies based on database state — some backend-specific:

1. PostgreSQL: `_is_collation_deterministic()` (postgresql/schema.py:369)
When creating indexes on text columns with collation, Django queries pg_collation to check if the collation is deterministic. Non-deterministic collations can't be used in plain indexes, so Django switches to CREATE INDEX CONCURRENTLY with an expression. The SQL differs based on catalog state.
2. MySQL: `_field_should_be_indexed()` (mysql/schema.py:122)
Queries information_schema for the storage engine. On InnoDB, Django skips creating FK indexes (InnoDB does it automatically); on MyISAM they're emitted explicitly. sqlmigrate against an InnoDB vs MyISAM copy would produce different CREATE INDEX output.
3. MySQL: `_create_missing_fk_index()` (mysql/schema.py:140)
When dropping composite indexes (via AlterUniqueTogether, AlterIndexTogether, RemoveIndex, RemoveConstraint), MySQL checks whether the FK column is "covered" by the index being removed. If so, it re-creates the implicit FK index. The output depends on which indexes actually exist on-disk.
4. Oracle: `_is_identity_column()` (oracle/schema.py:214)
When altering column types, checks user_tab_cols for identity columns. Affects whether DROP IDENTITY is emitted.
5. Oracle: `_get_default_collation()` (oracle/schema.py:240)
Queries user_tables for the table's default collation during collation changes, affecting COLLATE clause generation.
All of these are cases where sqlmigrate produces different SQL depending on which database you point it at, not just constraint name resolution. Would you like me to propose revised docs that generalize the principle and mention some of these backend-specific examples?

It seems all items are index related, so perhaps the documentation from the PR author already covers them sufficiently. I'll ask for confirmation.
